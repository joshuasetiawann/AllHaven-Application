# v0.16.0 — Persistent AI Memory

**Date:** 2026-06-12 · **Type:** Minor · **Archive:** `AllHaven 2.8` on `master`.

A full-stack feature release. Backend gets a new memory service, REST API, and DB
migration (`0007`); the frontend gets a memory management page, in-chat indicator, and
nav entry. All four chat modes now receive workspace memory as context.

## What changed

### Persistent workspace-scoped AI memory
- Memories are stored in the database and **survive restarts**, browser clears, and
  device switches (unlike the local-only per-section memory introduced in v0.15.0).
- **Auto-learning:** every assistant response is processed by a hybrid pipeline —
  a rule-based fast-path extracts definite facts synchronously, while an async
  background LLM pass extracts subtler context without blocking the chat reply.
- **Secret detection:** every memory candidate is scanned for API keys, tokens,
  bearer/JWT patterns, and similar secrets before writing. Flagged items are
  **quarantined** into the approval queue rather than stored directly.
- **Approval flow for sensitive memories:** the pending-approval queue surfaces
  borderline items for human review. Approve to persist, reject to discard.

### Context injection in all four chat modes
- Single-agent, multi-agent parallel, debate, and reasoning modes each receive the
  workspace memory as a **system-level context prefix** on every turn.
- Each injection is keyed by `section_key` (matches the section/module the user is in),
  so only the relevant slice of memory is included.
- The existing per-section local memory (v0.15.0) and the new persistent memory are
  layered: persistent memory provides background facts; per-section memory provides
  session-specific overrides.

### Memory tools in the Tool Registry
Five new tools registered alongside the existing 35+:

| Tool | Type | Approval required |
|------|------|------------------|
| `memory_list` | read | no |
| `memory_search` | read | no |
| `memory_create` | write | **yes** |
| `memory_update` | write | **yes** |
| `memory_delete` | write | **yes** |

Write operations follow the established Tool Registry pattern: the AI proposes the
action, and a human Approves / Edits / Rejects it in the chat Pending Actions panel.

### REST API `/ai/memory/*`
Full CRUD and search surface exposed to the frontend and any external tooling:

- `GET    /ai/memory/`          — list memories (paginated, filterable by section/tag)
- `POST   /ai/memory/`          — create a memory (bypasses approval; for manual entry)
- `GET    /ai/memory/pending`   — list memories awaiting approval
- `GET    /ai/memory/{id}`      — fetch one memory
- `PATCH  /ai/memory/{id}`      — update a memory
- `DELETE /ai/memory/{id}`      — delete a memory
- `POST   /ai/memory/search`    — semantic / keyword search across the store

### Frontend: memory management page
- New page at `/dashboard/ai/memory` — browse, search, approve, edit, and delete
  memories from the dashboard.
- Pending-approval items appear at the top with Accept / Reject controls.
- Memories are grouped by `section_key`; a search bar filters in real time.

### In-chat MemoryIndicator
- A compact indicator appears in the chat header whenever memory context is active.
- Clicking it navigates to the memory management page.
- Shows an error state when memory injection fails (e.g. DB unreachable) rather than
  silently dropping context.

### AI Memory nav link
- A dedicated **AI Memory** entry under the AI section of the sidebar links directly
  to `/dashboard/ai/memory`.
- The sidebar's longest-prefix highlight algorithm correctly marks it active vs.
  the parent `/dashboard/ai` page.

### Supabase background sync (opt-in)
- One-way sync pushes new and updated memories to a Supabase table in the background.
- Disabled by default. Enable by setting `SUPABASE_URL` and `SUPABASE_ANON_KEY` in
  `.env` (or `backend/.env`). No Supabase credentials → sync is a no-op; the rest
  of the memory system is unaffected.
- Sync events are recorded in `ai_memory_sync_log` for debugging.

## Database changes — migration required

This release adds **three new tables** via migration `0007_ai_memories`:

| Table | Purpose |
|-------|---------|
| `ai_memories` | Main memory store (content, section_key, source, approval state, timestamps) |
| `ai_memory_tags` | Many-to-many tag associations for memories |
| `ai_memory_sync_log` | Log of Supabase sync events (opt-in) |

**After pulling, run:**

```bash
cd backend
python -m alembic upgrade head
```

If you are running Docker, restart the backend container after migrating.

## Upgrade notes

1. Run `python -m alembic upgrade head` (see above).
2. Restart the backend.
3. Rebuild the frontend (`npm install && npm run build` in `frontend/`, or relaunch Haven).
4. Memory starts empty — it fills as you chat.
5. To enable Supabase sync, add `SUPABASE_URL` and `SUPABASE_ANON_KEY` to `.env` and
   restart the backend.

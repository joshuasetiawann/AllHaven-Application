# v0.9.0 — One-click desktop installer, setup wizard & System Control

**Date:** 2026-06-10 · **Type:** Minor (major feature) · **Archive folder:** `AllHaven 2.1` on `master`.

This release makes Haven **clone → click → run** for non-technical users, and adds
an in-app **System Control** panel to start/stop/restart services, edit ports, and
read (masked) logs — all behind a secure, localhost-only control agent.

---

## 1. Executive summary

- **Built:** OS launchers, a stdlib browser **setup wizard**, a localhost **control agent**, an authenticated **`/system/*` API**, and a **Settings → System Control** UI (service cards, ports editor, log viewer). Plus docs + tests.
- **Release-ready?** Yes for the code paths that can be verified without a Docker daemon/desktop session: pure logic, API behavior, allowlists, masking, `.env` safety, and builds all pass. The live Docker/desktop integration (actually launching containers, creating a real `.lnk`, opening a GUI) must be smoke-tested on a real machine — see §10–11.
- **Remaining risk:** low and contained. Privileged operations are isolated to a localhost+token agent with strict allowlists; the control surface is off outside local mode and requires auth.

## 3. Installation flow (clone → open Haven)

1. Clone the repo. 2. Open the launcher for your OS. 3. The **wizard** opens in your browser: OS → system check → Docker help (if needed) → choose ports → write `.env` → start services → health check → create desktop shortcut. 4. Click **Haven** (desktop) anytime → it ensures the agent + services are up and opens the app. 5. Manage services in **Settings → System Control**.

## 4. OS support

| OS | Launcher | Shortcut | Notes |
|----|----------|----------|-------|
| **Linux** | `.sh` | `Haven.desktop` (Desktop → fallback `~/.local/share/applications`) | Verified file generation; clicking needs a desktop session. |
| **macOS** | `.command` | `~/Desktop/Haven.command` | First open may need right-click → Open (Gatekeeper). |
| **Windows** | `.bat` | `Haven.lnk` via PowerShell, fallback `Haven.bat` | `.lnk`/PowerShell path needs a real Windows box to confirm. |

## 5. Docker handling

- **Detection:** `docker` on PATH (installed), `docker info` rc==0 (running), `docker compose version` (compose).
- **Missing:** the wizard does not crash — Step 4 shows OS-specific install guidance + a link, with **Check Again**.
- **Installed but not running:** wizard tells you to open Docker Desktop / start the service, then **Check Again**. The agent reports Docker services as `unavailable` until the daemon is up.

## 6. Port configuration

- **Defaults:** frontend 3000, backend 8000, Postgres 5432 (optional: redis 6379, n8n 5678, ollama 11434).
- **Conflict detection:** numeric/range validation, duplicate detection, and a real TCP check for "already in use", with a **free-port suggestion**.
- **Saved:** to `.env` via a dedicated atomic writer (timestamped backup, 0600). Changing the Postgres port re-derives `DATABASE_URL`. "Port changes require a service restart" — the editor offers **Save & Restart**.

## 7. Desktop shortcut

Created in Step 9 (or re-creatable). Points at the OS launcher; because `.env` now exists, clicking runs `haven_launch.py`: ensure agent → start Postgres/backend/frontend → wait for the frontend port → open the browser. If the agent can't start, it prints where to look (`var/logs/agent.log`).

## 8. System Control panel

- **Cards:** backend, frontend, postgres (+ optional n8n/ollama when present) with status, port, last-checked, and `message`.
- **Actions:** Start / Stop / Restart (host via the agent; Docker via Compose) and **Logs** (masked, last N lines). Disabled buttons explain why (agent offline / not controllable).
- **Ports editor:** edit + Save or Save & Restart, with the restart warning and inline validation.
- **Fallback:** if the agent is offline, status is still shown (port-based) read-only, with a clear banner.

## 9. Security fixes & posture

**New surface, secured by design:**
- Privileged ops only in the **localhost-only (127.0.0.1)**, **token-gated** (0600 token, constant-time compare) agent.
- **Allowlists** for services and actions enforced in **both** the agent and the backend.
- **No shell anywhere** — `subprocess` with fixed argv lists; Docker via a builder that **cannot** produce `down`/`rm`/volume operations.
- **Secret masking** applied to all logs/errors returned (keys, URL creds, bearer tokens) — covered by tests.
- **`.env` safety** — atomic write + timestamped backup + 0600; only a fixed key set is written; existing secrets preserved; `.env.bak.*` is git-ignored.
- **No DB creds in the browser**; the ports API never returns `DATABASE_URL`. `DATABASE_URL` remains excluded from the web-sync allowlist (unchanged).
- **AuthN/Z:** every `/system/*` endpoint requires an authenticated principal; control is **disabled outside local mode** (`is_local_env`), so production deployments don't expose it.
- **Path traversal:** log/service identifiers are validated against the allowlist before any file/compose access.

**Remaining recommendations:** add a desktop-session smoke test in CI matrices per OS; consider signing the Windows `.lnk`/SmartScreen guidance; optional rate-limit on `/system/*`.

## 11. How to test manually

1. **Fresh clone:** clone, open the launcher → wizard opens at `http://127.0.0.1:7000`.
2. **First install:** complete the wizard with Docker running → health all green → app opens.
3. **Docker missing:** quit Docker → wizard Step 4 shows guidance + Check Again (no crash).
4. **Port conflict:** set frontend to a used port → Step 5 flags it + suggests a free one.
5. **Restart frontend / backend:** Settings → System Control → Restart → status returns to running; logs show the new process.
6. **Stop/Start services:** Stop Postgres → card shows stopped → Start → running.
7. **Desktop shortcut:** click the Haven icon while services are stopped → it starts them and opens the app.
8. **Logs:** open a service's logs → confirm content shows and any secret-like values are masked.

## 12. Known limitations

- **Host self-control:** the backend cannot restart *itself* from within its own request; the agent owns that, so backend Start/Stop/Restart require the agent to be running (clear fallback shown otherwise).
- **Windows status checks** rely on `os.kill(pid, 0)` + port probing (no `psutil`); robust but best-effort.
- **Ollama** is shown as status-only (we never start/stop the user's host daemon). **Redis/n8n** appear only when enabled in compose.
- Docker/desktop paths verified by logic + unit tests, not a live end-to-end run in CI (no daemon/GUI in the build sandbox).

# v0.10.0 — Reliable one-click startup + responsive menu

**Date:** 2026-06-10 · **Type:** Minor · **Archive:** `AllHaven 2.2` on `master`.

Two things: **fix the "starts manually but not from the app" bug** so Haven runs
directly from the launcher/wizard, and make the **navigation menu nicer and more
responsive**.

---

## 1. The bug — why it worked manually but not from the app

Running manually (`allhaven.sh run`) works because it: creates `frontend/.env.local`,
starts Postgres and **waits** for it, ensures the venv + `npm install`, runs
`alembic upgrade head`, and starts the backend on **`0.0.0.0`**.

The v0.9.0 app launch path skipped most of that and bound the backend to
**`127.0.0.1`**. The result was a backend that either never came up or wasn't
reachable at the URL the frontend used (a `localhost` lookup can resolve to IPv6
`::1` and miss an IPv4-only bind), so the login screen showed *"Cannot reach the
AllHaven API."*

## 2. The fix — make the app path faithful to `allhaven.sh`

All launch logic now mirrors the proven manual flow and is shared by the desktop
shortcut, the wizard, and in-app restarts:

- **Bind `0.0.0.0`** for the backend & frontend (the agent itself stays
  `127.0.0.1`-only). `installer/haven_common.py: APP_BIND_HOST`.
- **Create `frontend/.env.local`** from its example if missing (`ensure_env_files`).
- **Order + wait + health-gate:** `installer/haven_launch.py` starts Postgres and
  waits for it, then starts the backend and **waits for `/api/v1/health`**, then the
  frontend, then opens the browser.
- **Run migrations** (`alembic upgrade head`) before the backend starts (in the agent).
- **Install missing deps on first run:** venv + `pip install -r requirements.txt`
  and `npm install` (idempotent; skipped when already present).
- **Enrich `PATH`** so GUI-launched processes find `node`/`npm`/`docker`/`python`
  (`enriched_env`) — the classic "works in terminal, not when double-clicked" cause.
- **Surface failures:** a masked tail of `var/logs/backend.log` / `frontend.log` is
  printed instead of failing silently; the agent returns a clear message if a
  service isn't set up.
- The wizard's **Start** step and the desktop shortcut both now drive this one
  launcher path (`/api/launch` → `haven_launch.py`).

**Files:** `installer/haven_common.py`, `installer/haven_agent.py`,
`installer/haven_launch.py`, `installer/haven_setup.py`, `installer/web/index.html`.

## 3. Responsive, polished navigation menu

`frontend/components/layout/{AppShell,Sidebar,Topbar}.tsx`:

- **Collapsible desktop rail** (260px ↔ 76px) with a persisted preference
  (`localStorage`), icon tooltips, and content padding that tracks the width.
- **Better mid-size behavior:** a persistent **icon rail on tablets (`md`+)**, full
  sidebar at `xl`; the mobile drawer remains for `< md`. The Topbar hamburger now
  hides exactly when the rail appears (no double-nav).
- **Polish:** a signed-in **user chip** (avatar + email), refined active/hover
  states, `aria-current` + focus rings, consistent dark-theme tokens.

## 5. How to verify on your machine

1. Ensure Docker is running. Delete `frontend/.env.local` and `backend/.venv` to
   simulate a fresh machine (optional).
2. Open the launcher for your OS. First run installs dependencies (a few minutes)
   — watch the wizard's **Health** step (or the launcher console / `var/logs/setup.log`).
3. The app should open and the login screen should reach the API (no "Cannot reach"
   error).
4. In **Settings → System Control**, Restart the backend and confirm it returns to
   healthy; open its **Logs** to see the `alembic upgrade head` + uvicorn lines.
5. Resize the window: full sidebar at wide widths, icon rail on tablet, hamburger
   drawer on mobile; toggle the desktop collapse and reload to confirm it persists.

## 6. Known limitations

- Changing the **backend port** later doesn't auto-update `frontend/.env.local`'s
  `NEXT_PUBLIC_API_BASE_URL` (defaults are handled). Use the default 8000 or update
  `.env.local` if you change it — a follow-up can wire this through System Control.
- First-run dependency install needs network access for `pip` / `npm`.

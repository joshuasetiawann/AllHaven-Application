# v0.12.0 — App-wide AI tools with human approval, 6 OpenRouter slots, 7-agent roles

**Date:** 2026-06-11 · **Type:** Minor (major feature) · **Archive:** `AllHaven 2.4` on `master`.

This release implements the AI core of the release spec: a safe **Tool Registry**
connecting AI Chat to every module, a working **human approval system**, **six
OpenRouter model slots**, **2 model slots per other provider**, **up to 7 agents**
with distinct roles, a **debate-flow visibility toggle**, and **output-quality
polish** — plus the Settings surfaces for all of it.

---

## 1. Executive Summary

- **Built:** backend Tool Registry (35 allowlisted tools) + orchestrator tool loop
  + approve/edit/reject execution; `openrouter_4..6` (12 providers); model slots;
  7-agent cap with roles; workspace chat-behavior settings; synthesizer prompt
  upgrade; Chat UI (agent selector ×7, tool activity, pending-actions panel,
  debate toggle) and Settings tabs (AI Tools, AI Chat, slots editor).
- **Release-ready?** Yes for everything verifiable without live model keys:
  168 backend tests pass (13 new covering the security core), frontend type-checks
  and builds. The **live** tool loop needs a real OpenAI-compatible key to
  smoke-test end-to-end (steps in §17).
- **Remaining risk:** low. Writes cannot execute without approval (HIGH-risk
  never auto-executes); tools/services/actions are allowlisted; nothing trusts
  raw model output.

## 8. Local Haven Agent

Unchanged (localhost-only, token-gated, allowlisted). New in this release: the AI
can *propose* `restart/start/stop_service` through the registry — those are
HIGH-risk tools, so they **always** require human approval, then execute through
the same backend `system_service` → agent path used by System Control.

## 9. System Control Panel

Unchanged UI. Now also reachable by AI tools: `get_service_status` and
`get_service_logs` (read, masked) run instantly; control actions require approval.

## 10. AI Integration (modules ↔ tools)

| Module | Read tools (auto-execute) | Write tools (approval) |
|---|---|---|
| Time | get_current_time | — |
| Tasks | list_tasks | create/update/complete/delete_task |
| Calendar | list_events | create/update/delete_event |
| Notes | list_notes, search_notes | create/update/delete_note |
| Finance | list_transactions, finance_monthly_summary, list_finance_categories | create/update/delete_transaction |
| Files | list_files, search_files (metadata only) | delete_file (HIGH) |
| Weather | get_current_weather (honest `setup_required`) | — |
| Automations | list_workflows | create/update_workflow_draft, enable_workflow (HIGH), disable_workflow |
| System | get_service_status, get_service_logs (masked) | restart/start/stop_service (HIGH) |

Natural-language examples that now work (with a tool-capable provider):
"What's my schedule today?", "Create a task for tomorrow at 9 AM" (→ pending
approval), "How much did I spend this month?", "Restart backend service"
(→ pending approval). Context policy: the orchestrator sends recent conversation
history only; data enters the prompt exclusively through tool results — never a
database dump.

## 11. AI Provider System

- **OpenRouter ×6** (`openrouter_1..6`): independent key/model/status each, with
  suggested roles (Main, Reasoning/Planner, Critic/Reviewer, Coding/Technical,
  Research/Long Context, Synthesizer/Final Writer). Env keys
  `OPENROUTER_{1..6}_API_KEY` / `_DEFAULT_MODEL`.
- **Other providers ×2 slots** (Ollama, GPT, Claude, Gemini, Grok, Blackbox):
  slot 1 = default model; slot 2 = optional secondary model + role, stored in the
  provider's `public_config` and selectable in chat as `provider#2`. Unconfigured
  /disabled slots are skipped with honest statuses.
- No model names are hardcoded as requirements — users configure all models;
  empty slots never break the system. Keys remain server-side, encrypted,
  masked in the UI; "Test Connection" honesty unchanged.

## 12. Multi-Agent System

- Cap raised to **7** (schema + service + UI). >7 → HTTP 422.
- Distinct default roles by position: Main Assistant, Planner, Research/Context,
  Technical/Coder, Critic/Risk, Product/UX, Synthesizer — overridden by slot
  roles; recorded in run/message meta and shown as badges.
- **Debate visibility**: workspace setting `show_debate_flow`. ON = full
  rounds/critique/synthesis transcript (existing behavior, still the default);
  OFF = only the polished final answer + "N agents collaborated". No hidden
  chain-of-thought is ever exposed — only the per-agent contributions the app
  already stores.
- One agent failing never fails the run (unchanged); unavailable models keep
  honest statuses (`not_configured`/`disabled`/`blocked`).

## 13. Human Approval System

- **Default ON** (`require_approval`). Read tools run instantly; write tools
  create `PENDING` proposals. **HIGH-risk tools require approval even when the
  workspace turns approvals off.**
- Pending actions panel (AI Chat): tool, risk badge, payload preview, **Approve**
  (executes through the registry → real result returned), **Edit** (JSON payload,
  validated), **Reject** (does nothing). Double-approve rejected; failed
  execution after approval is reported honestly (`422 — execution failed: …`).
- Every call (executed / pending / approved / rejected / failed) is written to
  the append-only audit log with tool, args, and status.
- The model is explicitly instructed — and the registry result enforces — that a
  pending action is **not** executed; it can never claim otherwise truthfully.

## 14. Output Quality Improvements

Debate + Reasoning synthesizer prompts now require: direct answer first; concrete
and specific; no generic filler/repetition; resolve disagreements with a stated
position; preserve warnings & security concerns; honest about uncertainty and
missing data; next steps when actionable; **answer in the user's language**
(Indonesian → natural Indonesian). The single-agent system prompt applies the
same policy and the strict tool honesty rules.

## 15. Security Fixes / Posture (new surface)

- Tool names, service names, and actions: **fixed allowlists**; unknown anything
  → rejected and audited. Arguments validated (Pydantic schemas / explicit
  parsing); arbitrary shell/SQL is not expressible.
- AI never touches DB/files/shell directly — only module services that enforce
  workspace scoping; file tools expose metadata only (no raw paths).
- Writes gated by approval (HIGH always); proposals are workspace-scoped;
  edit/approve/reject require auth.
- Logs returned by tools are masked (existing `system_service` masking); secrets
  never enter tool results; provider keys stay encrypted server-side.
- Model output is parsed, never executed: tool requests go through the registry;
  malformed JSON arguments become honest tool errors.
- CORS posture unchanged (documented local-mode choice from v0.7.0).

## 17. Manual Test Steps

1. Configure an OpenAI-compatible provider (OpenAI key, or an OpenRouter agent)
   in Settings → AI Providers; enable it.
2. AI Chat (single agent): ask "What time is it?" → tool chip `get_current_time ·
   executed`, correct answer. Ask "Create a task to call Owen tomorrow 9am" →
   pending action appears; the reply says it awaits approval; Tasks unchanged.
3. Approve it in the Pending actions panel → task appears in Tasks. Create
   another, **Edit** the payload title, approve → edited title. Reject one →
   nothing happens.
4. Ask "How much did I spend this month?" → finance summary from real data.
5. Settings → AI Tools: disable `list_tasks`, ask "what are my tasks" → honest
   "tool disabled" behavior; re-enable.
6. Settings → AI Chat: turn **Show debate flow** off; run a Debate with 3+
   agents → only the final answer + "N agents collaborated". Toggle back on →
   full transcript.
7. Select **7 agents** (e.g. 6 OpenRouter + Ollama) → run completes; an 8th
   can't be selected; API returns 422 for 8.
8. Settings → AI Providers → Claude: set Slot 2 model + role, then select
   "Claude · Slot 2" in chat alongside slot 1 — two Claude agents with different
   models/roles.
9. Ask "Restart backend service" → HIGH-risk pending action; approve → service
   restarts via the local agent (System Control shows it).

## 18. Known Limitations

- **Native tool calling** works on the OpenAI-compatible family (OpenAI, Grok,
  OpenRouter ×6). Anthropic, Gemini, Ollama, and Blackbox chat (with history)
  but without tools this release — they say nothing false about tools.
- Multi-agent runs don't run the tool loop (single-agent does); agents get role
  prompts + the question. Spec'd as a next step.
- `ai_model_slots` / chat & tool settings persist as reserved
  `integration_configs` rows and proposals/audit reuse existing tables — a
  deliberate no-migration design (documented here) instead of the spec's new
  table list.
- Not yet implemented (honest): `get_note`/`summarize_file`, file move/folder
  tools, weather forecast tool, n8n `run_workflow`, an audit-log viewer in
  Settings → AI Tools.

# AllHaven 3.7.0

Release date: 2026-06-19

## Summary

AllHaven 3.7 is the **two-way Supabase sync + mobile-on-Supabase** release.
Desktop keeps its local-first PostgreSQL as the source of truth and now gains a
real **incremental two-way sync engine** that keeps local Postgres and the
Supabase project converged in the background (Last-Write-Wins by `updated_at`,
soft-delete tombstones, anti-loop / echo-suppression, visible sync health at
`GET /settings/sync/status`). This replaces the old one-way push mirror from
3.4.

On the mobile side, a new Supabase-native data layer lets the Android APK talk
**directly to Supabase** — Supabase Auth handles login and Row Level Security
gates every read/write, so mobile works with zero AllHaven backend in the path.
The data mode is selected at APK build time via `NEXT_PUBLIC_DATA_MODE=supabase`.

## Added

- **Two-way incremental sync engine.** A background sync loop compares local
  Postgres and Supabase by `updated_at` watermark, merges on Last-Write-Wins,
  and writes the winning row to whichever side lost — all without looping the
  update back as a spurious change. Deleted rows travel as soft-delete tombstones
  (`deleted_at`) so nothing "resurrects" across the sync boundary.
  `GET /settings/sync/status` exposes per-table lag and overall health.

- **Mobile Supabase data layer.** New client-side data access objects for Tasks,
  Notes, Finance, Calendar, Routines, Automations, Weather locations, and the
  Dashboard query the Supabase PostgREST API directly. The mobile APK no longer
  needs a reachable AllHaven backend.

- **Supabase Auth on mobile.** Mobile login now authenticates with Supabase Auth
  (email + password). Session tokens are stored in the mobile keychain / secure
  storage and refreshed automatically. Web / desktop keep their existing
  cookie + bearer flow.

- **Supabase Auth provisioning.** New user accounts are provisioned in Supabase
  automatically via service-role on signup (best-effort, never blocks
  registration). Existing accounts that predate 3.7 can link their Supabase
  identity through a new **Connect to Supabase** button in Settings → Account
  (`POST /settings/supabase/connect`).

- **Database migrations 0010 – 0015:**
  - `0010`: `deleted_at` soft-delete column on all synced tables.
  - `0011`: `profiles.supabase_user_id` identity-mapping column.
  - `0012`: DB-authoritative `updated_at` trigger (Postgres sets the timestamp,
    never the application layer).
  - `0013`: Supabase Row Level Security — workspace-scoped policies on all
    synced tables using `app_user_id()` / `is_member()` helper functions (applied
    only when `ALLHAVEN_DB_TARGET=supabase`).
  - `0014`: `workspace_members` RLS hardening — SELECT restricted to own row;
    INSERT/UPDATE/DELETE restricted to service_role. Closes a
    privilege-escalation path where any workspace member could mutate membership
    rows.
  - `0015`: `sync_state` watermark table — persists the per-table high-water
    mark for the incremental sync loop.

## Fixed

- **Login / dashboard request timeout.** The initial session check and dashboard
  data fetch now time out after a configurable interval instead of waiting
  forever, eliminating the infinite spinner when the backend is stalled or
  unreachable.

- **Checklist item soft-delete.** Deleting a checklist item on desktop now
  writes `deleted_at` instead of hard-removing the row. This prevents a
  completed item from "resurrecting" on the next sync cycle when Supabase still
  holds the original row.

- **Checklist item ordering.** Ordering is now stable across desktop ↔ mobile
  sync; items sorted by `position` on one side maintain the same order on the
  other.

## Deferred

The following are scoped to v3.8 / v3.9:

- AI agents and third-party integrations (secrets stored in Supabase Vault).
- Automation execution via Supabase Edge Functions.
- Drive (file storage) migrated to Supabase Storage.
- Knowledge / RAG on mobile via Edge Functions.

## Upgrade notes

**Backend (desktop):**

```
python -m alembic upgrade head
```

**Supabase database** (RLS migrations):

```
ALLHAVEN_DB_TARGET=supabase python -m alembic upgrade head
```

Migrations 0010 – 0012 are safe for both targets. Migration 0013 and 0014 apply
Supabase-specific RLS policies and are skipped (no-op) when
`ALLHAVEN_DB_TARGET` is not `supabase`.

**Mobile APK build** — the following CI variables must be set on the repo before
triggering `android-apk.yml`:

| Name | Kind | Notes |
|---|---|---|
| `SUPABASE_URL` | Variable | Project API URL from the Supabase dashboard |
| `SUPABASE_ANON_KEY` | Secret | `anon` / `public` key for client-side use |

No changes to the desktop startup command or frontend build for web/desktop
deployments.

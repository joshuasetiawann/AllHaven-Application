# v0.4.0 — Image input (vision) & polished chat output

**Date:** 2026-06-09
**Type:** Minor (new feature)
**Archive folder:** `AllHaven 1.5` on the `master` branch.

Two improvements to the AI chat: it now reads cleanly, and it can see images.

## Polished chat output (Markdown)
AI replies are rendered as Markdown — headings, ordered/unordered lists, fenced
code blocks, inline code, bold/italic, blockquotes, and links — instead of being
shown as one raw, run-on blob. The renderer is small and dependency-free and never
uses `dangerouslySetInnerHTML`, so there is no HTML-injection risk.

Applied across the single bubble, the per-agent debate cards, and the synthesized
final answer.

## Image upload + vision
- Attach up to **4 images** per chat turn from the composer (paperclip button).
- Images are **downscaled client-side** (max 1280px, JPEG) so the payload stays small.
- The image is sent with the message and **formatted per provider**:
  - OpenAI / OpenRouter / Grok / Blackbox — `image_url` content parts
  - Anthropic — base64 `image` blocks
  - Gemini — `inline_data` parts
  - Ollama — base64 `images` array (for vision models such as `llava`)
- Works in **Parallel**, **Debate** (round 1), and **Reason** (Analyst) modes.
- The attached image is **persisted** on the user message and shown again on reload.

> Vision requires a vision-capable model (e.g. `gpt-4o`, `claude-3.5`, `gemini-1.5`,
> `llava`). A non-vision model will say so honestly rather than pretend.

## Fixed
- Missing `network_error_message` import in the Ollama, Anthropic, and Gemini
  adapters — a network failure during chat would have raised `NameError` instead
  of returning an honest "could not reach the provider" message.

## Tests & verification
- `tests/test_ai_images.py`: data-URL parsing, OpenAI image-part formatting, and an
  end-to-end check that an attached image reaches the provider and is persisted.
- Backend suite: **105 passing**. Frontend: typecheck + production build pass.

## Known limitations
- Attached images are stored as data URLs on the message metadata; very heavy image
  use will grow the database. (A future version can offload to the Drive module.)
